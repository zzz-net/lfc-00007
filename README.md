# 值班交接白板

一个本地运行的运维班组值班交接白板系统，支持事项管理、交接班确认、历史记录查询等功能。

## 功能特性

- 📋 **看板式管理**：五列看板（新建/处理中/待接班确认/已接收/已关闭）直观展示事项状态
- ⚙️ **灵活配置**：支持班次、角色、用户、交接模板和检查项的自定义配置
- 🔄 **事项流转**：完整的状态流转流程，每次操作都有历史记录
- 🛡️ **权限控制**：基于角色的权限管理，不同角色有不同操作权限
- 📝 **版本控制**：乐观锁机制，两次基于同一版本的编辑会提示冲突
- ✅ **必填校验**：缺少必填检查项时不能进行接班确认
- 📤 **交接单导出**：支持导出文本格式的交接单
- 📜 **历史记录**：支持按班次、日期范围筛选历史交接记录
- 💾 **备份与恢复中心**：支持完整数据包的 JSON 导出和导入恢复
  - 导出内容包含：班次、角色、用户、检查项、事项、交接记录、恢复日志、复盘卡、版本号和导出时间
  - 导入前校验：结构校验、版本兼容检查、权限校验（仅配置管理员可恢复）
  - 恢复预览：展示导入包摘要、当前数据摘要、差异数量和冲突提示
  - 冲突检测：同名班次、同名用户、未关闭事项编号冲突
  - 恢复日志：完整记录每次恢复操作（成功/失败）、操作人、数据快照、冲突信息
- 🔍 **交接复盘**：对已关闭事项和交接记录生成复盘卡，追踪遗留风险和后续跟进
  - 班长可创建复盘卡，标记遗留风险、责任人、跟进截止时间和复盘结论
  - 普通值班员可查看复盘卡和补充跟进说明，但不能修改结论或删除复盘卡
  - 复盘卡自动带出原始摘要，所有修改写入可读日志
  - 复盘信息随导出/恢复持久化，刷新页面不丢失
- 🔍 **日常追踪**：班长可在复盘中心按条件快速筛选和标记重点关注复盘卡
  - 筛选条件：有遗留风险、无风险、只看我负责、已逾期
  - 重点关注标记：班长可将复盘卡标为重点关注，卡片和详情中稳定显示标记
  - 筛选条件和重点标记跟随 localStorage 保存，刷新页面不丢失
  - 完整 JSON 导出/恢复后重点标记和筛选条件完整带回
  - 恢复日志显示复盘卡数量对比，便于确认数据无异常变化
  - 普通值班员可查看筛选结果和重点标记，但不能修改重点状态
  - 班长修改重点状态时自动写入复盘卡操作日志
- 💾 **本地持久化**：使用 localStorage 存储，重启后数据不丢失

## 数据模型

### 班次 (Shift)
- id: 唯一标识
- name: 班次名称
- startTime: 开始时间 (HH:mm)
- endTime: 结束时间 (HH:mm)

### 角色 (Role)
- id: 唯一标识
- name: 角色名称
- canCreate: 可创建事项
- canProcess: 可处理事项
- canClose: 可关闭事项
- canConfirm: 可接班确认

### 用户 (User)
- id: 唯一标识
- name: 用户姓名
- roleId: 所属角色ID

### 检查项 (CheckItem)
- id: 唯一标识
- name: 检查项名称
- required: 是否必填
- description: 检查说明

### 事项 (Item)
- id: 唯一标识
- title: 事项标题
- type: 类型（alert告警/change变更/other其他）
- description: 事项描述
- status: 状态（new/processing/pending_confirm/received/closed）
- version: 版本号（用于冲突检测）
- shiftId: 所属班次
- shiftDate: 班次日期
- creatorId/creatorName: 创建人
- assigneeId/assigneeName: 处理人
- createTime/updateTime/closeTime: 时间戳
- closeReason: 关闭原因
- history: 操作历史数组

### 交接记录 (HandoverRecord)
- id: 唯一标识
- shiftId/shiftName: 班次信息
- date: 交接日期
- handoverUserId/handoverName: 交班人
- takeoverUserId/takeoverName: 接班人
- handoverTime/takeoverTime: 时间
- status: 状态（pending/confirmed）
- itemIds/itemCount: 关联事项
- checkResults: 检查项结果
- itemsSnapshot: 事项快照

### 恢复日志 (RecoveryLog)
- id: 唯一标识
- action: 操作类型（数据恢复/导入失败）
- operatorId/operatorName: 操作人
- importVersion: 导入数据版本
- importTime: 导入数据导出时间
- beforeSummary: 恢复前数据摘要
- afterSummary: 恢复后数据摘要
- conflicts: 冲突信息（班次、用户、事项）
- reason: 失败原因（仅失败时）
- succeeded: 是否成功
- timestamp: 操作时间戳

### 第一步：创建样例数据

1. 打开 `index.html` 页面
2. 点击右上角「⚙️ 配置」按钮进入配置页

6. 系统会自动跳回看板页面
7. ✅ **预期结果**：页面顶部显示「白班」班次，看板上有3个样例事项

### 第二步：提交事项

1. 确保顶部当前用户选择了「张三（班长）」
2. 在看板页面点击「+ 新建事项」按钮
3. 填写事项信息：
   - 标题：测试告警事项
   - 类型：未处理告警
   - 描述：这是一个测试用的告警事项

4. 点击「创建事项」按钮
5. ✅ **预期结果**：

### 第三步：事项处理流转

1. 点击「新建」列中的「数据库CPU使用率告警」事项
2. 在详情弹窗中点击「🔄 开始处理」按钮
3. ✅ **预期结果**：
   - 事项状态变为「处理中」
   - 版本号从 v1 变为 v2

7. ✅ **预期结果**：事项卡片移动到「待接班确认」列

### 第四步：补齐检查项并接班确认

### 第六步：版本冲突测试

> 注意：此测试需要在浏览器开发者工具中进行，或者通过快速连续操作模拟。

1. 选择「张三（班长）」作为当前用户
2. 打开一个「处理中」的事项详情，记住当前版本号（如 v3）
3. 打开浏览器开发者工具（F12），切换到 Console 标签
4. 执行以下命令手动模拟冲突场景：
   ```javascript
   const item = Store.Items.getAll()[0];
   console.log('当前版本:', item.version);
   const result = Store.Items.update(item.id, { title: item.title + ' (其他人修改)' }, {id:'user_li',name:'李四'}, item.version);
   console.log('李四修改结果:', result.success, '新版本:', result.item?.version);
   const result2 = Store.Items.update(item.id, { title: item.title + ' (张三修改)' }, {id:'user_zhang',name:'张三'}, item.version);
   console.log('张三修改结果(基于旧版本):', result2.success, '冲突:', result2.conflict);
   ```
5. ✅ **预期结果**：
   - 李四的修改成功，版本号递增
   - 张三基于旧版本的修改失败，返回 `conflict: true`
   - 张三的修改不会覆盖李四的修改
6. 刷新页面后再查看该事项
7. ✅ **预期结果**：事项标题是李四修改后的内容，没有被覆盖

**场景 3：Store 层幂等拦截（极端情况兜底，即使前端漏了也防重）**

此步骤在浏览器的开发者工具 Console 中执行，模拟前端由于网络延迟、快速连击、并发标签页等造成的重复调用：

```javascript
const adminUser = Store.Users.getById('user_zhang');

// 第 1 次调用 - 本来就有一张卡
const r1 = Store.ReviewCards.create({
  sourceType: 'closed_item',
  sourceId: '事项ID',
  sourceSummary: '试图重复创建',
  conclusion: '试图覆盖原有结论'
}, adminUser);

// 第 2 次调用 - 间隔 0ms，模拟快速连击
const r2 = Store.ReviewCards.create({
  sourceType: 'closed_item',
  sourceId: '事项ID',
  sourceSummary: '再次试图覆盖',
  conclusion: '结论2'
}, adminUser);

console.log('第1次结果：', { created: r1.created, sameId: true /* 原本就是首次或复用 */, count: Store.ReviewCards.getAll().length });
console.log('第2次结果：', { created: r2.created, message: r2.message, sameIdAsFirst: r2.card.id === r1.card.id, count: Store.ReviewCards.getAll().length });
```

3. ✅ **预期结果**（观察 Console 输出）：
   - `r1.created === false`（因为该事项原本已复盘，首次被调 create 也应拦截）
   - `r2.created === false`
   - `r1.card.id === r2.card.id`（两次都返回同一张卡的 id，不生成新卡）
   - `Store.ReviewCards.getAll().filter(c => c.sourceId === '事项ID').length === 1`（同一个 sourceId 下始终只有 1 张）

## 核心技术实现

### 数据持久化
使用浏览器 `localStorage` 进行数据存储，所有数据以 JSON 格式序列化存储。

### 版本冲突检测
采用乐观锁机制，每次修改时携带基础版本号，如果与当前版本不符则拒绝修改并返回冲突信息。

### 权限控制
基于角色的访问控制（RBAC），每个角色有独立的权限标志，操作前校验当前用户角色是否具备相应权限。

### 状态流转
事项状态流转图：
```
新建 → 处理中 → 待接班确认 → 已接收
 ↓        ↓          ↓          ↓
  --------- 可随时关闭 --------→ 已关闭
```

## 常见问题

**Q: 数据存储在哪里？会丢失吗？**
A: 数据存储在浏览器的 localStorage 中。只要不清除浏览器数据，数据就会一直保留。

**Q: 可以在多台设备间同步数据吗？**
A: 当前版本仅支持本地存储，不支持多设备同步。如需同步功能，需要后端服务支持。

**Q: 如何备份数据？**
A: 点击顶部导航栏的「💾 备份恢复」按钮进入备份与恢复中心，点击「📥 导出完整数据包」按钮即可下载完整的 JSON 格式备份文件。恢复时选择本地备份文件，经过校验和预览后即可恢复数据。只有具备配置管理权限的用户（如班长）才能执行恢复操作。

**Q: 支持多用户同时在线编辑吗？**
A: 不支持，本应用是单机版应用。但内置了版本冲突检测机制，防止同一事项被覆盖修改。
